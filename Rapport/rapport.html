2. Conception du Projet

2.1 Structure du Projet

PIL1_2526_37/
├── backend/
│   ├── messagerie.py       # API Flask + WebSocket (SocketIO)
│   ├── requirements.txt    # Dépendances Python
│   └── .env.example        # Modèle de configuration
├── frontend/
│   └── messagerie.html     # Interface utilisateur de la messagerie
├── Database/
│   └── Files.sql           # Schéma SQL + données de test
├── Rapport/
│   └── rapport.html        # Ce rapport
├── .gitignore
└── .env.example

Le projet suit une architecture client-serveur :

  • Backend (Flask) : API REST pour les discussions, messages et notifications. Communication temps réel via Flask-SocketIO.
  • Frontend (HTML/JS) : Interface utilisateur avec Socket.IO client pour la messagerie en direct.
  • Base de données (MySQL) : Stockage des utilisateurs, annonces, messages, notifications et matchings.

2.2 Base de Données

La base IFRI_MentorLink est constituée de 15 tables :

Table Rôle
Utilisateurs Profils des étudiants (nom, email, filière, niveau, bio)
Matières Liste des disciplines (Algorithmique, Python, BDD, etc.)
Annonces Offres et demandes de mentorat
Disponibilités Créneaux horaires (jour, heure début/fin)
Discussions Sessions de conversation entre utilisateurs
Messages Messages textes avec statut lu/non lu
Notifications Alertes (message, match, réponse)
Réponses Réponses aux annonces avec statut
Matching Paires mentor/mentoré avec score
MAITRISE Liaison utilisateur → matière maîtrisée
BESOIN Liaison utilisateur → matière souhaitée
USER_DISPONIBILITE Liaison utilisateur → créneaux
ANNONCE_MATIERE Liaison annonce → matière
ANNONCE_DISPONIBILITE Liaison annonce → créneau
PARTICIPATION Liaison utilisateur → discussion

Le schéma inclut des contraintes d'unicité pour éviter les doublons (réponses multiples à une annonce, paires mentor/mentoré en double) et des données de test (6 utilisateurs, 2 discussions, messages, notifications) pour valider le fonctionnement.

3. Instructions de Déploiement

Prérequis

  • Python 3.10+
  • MySQL 8.0+
  • Navigateur web moderne

Étapes d'installation

1. Cloner le dépôt
git clone <https://github.com/Christopher-arc/PIL1_2526_37.git> PIL1_2526_37
cd PIL1_2526_37
2. Créer la base de données
mysql -u root -p < Database/Files.sql
3. Créer l'environnement virtuel et installer les dépendances
python -m venv .venv
.\.venv\Scripts\Activate.ps1 # Windows
pip install -r backend/requirements.txt
4. Configurer les variables d'environnement
Copy-Item .env.example .env # Windows

Éditer le fichier .env avec vos identifiants MySQL :

DB_HOST=localhost
DB_USER=root
DB_PASSWORD=votre_mot_de_passe
DB_NAME=ifri_mentorLink
5. Lancer le serveur
python backend/messagerie.py

Le serveur démarre sur http://localhost:5001.

6. Ouvrir l'application

Naviguer vers http://localhost:5001 dans le navigateur.

4. Manuel d'Utilisation

4.1 Messagerie

La page principale de messagerie se compose de deux zones :

  • Panneau gauche : Liste des conversations triées par date, avec le dernier message et le nombre de messages non lus.
  • Panneau droit : Fenêtre de chat affichant l'historique des messages de la conversation sélectionnée.

Fonctionnalités :

  • Consulter une conversation : Cliquer sur une conversation dans la liste pour charger son historique.
  • Envoyer un message : Saisir le texte dans le champ en bas et appuyer sur Entrée ou cliquer sur le bouton ➤.
  • Messages en temps réel : Les nouveaux messages apparaissent instantanément grâce à SocketIO.
  • Marquage automatique : Les messages sont marqués comme lus dès que la conversation est ouverte.
  • Recherche : Filtrer les conversations par nom dans la barre de recherche.
  • Indicateur de non-lus : Un badge bleu sur une conversation indique les messages non lus.

4.2 Notifications

  • Cliquer sur l'icône 🔔 dans la barre de navigation pour ouvrir le panneau des notifications.
  • Les notifications sont classées par type (message, match, réponse) avec un code couleur.
  • Le badge rouge sur l'icône indique le nombre de notifications non lues, mis à jour automatiquement toutes les 3 secondes.

4.3 Comportement attendu

  • L'utilisateur courant est défini par défaut à l'ID 1 (Aimé Koffi) via la session Flask.
  • Les données de test incluent 2 discussions actives avec messages et notifications préremplis.
  • Le serveur WebSocket permet une communication bidirectionnelle sans rechargement de page.

4.4 Dépannage

Problème Cause possible Solution
Conversations vides Base de données non initialisée Exécuter Files.sql dans MySQL
WebSocket ne se connecte pas Port 5001 bloqué Vérifier le pare-feu ou changer de port
Erreur de connexion MySQL .env mal configuré Vérifier DB_HOST, DB_USER, DB_PASSWORD dans .env

4.5 Routes de l'API

Méthode Route Description
GET / Page HTML de la messagerie
GET /discussions Liste des conversations de l'utilisateur
GET /discussion/<id> Historique des messages d'une conversation
POST /discussion/<id>/read Marquer les messages comme lus
GET /notifications Notifications non lues de l'utilisateur
GET /notifications/count Nombre de notifications non lues
SocketIO send_message Événement d'envoi de message temps réel
SocketIO receive_message Réception d'un message temps réel
